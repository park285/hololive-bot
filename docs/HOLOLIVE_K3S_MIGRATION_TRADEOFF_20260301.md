# Hololive-bot Docker Compose → k3s 마이그레이션 트레이드오프 분석

- 작성일: 2026-03-01 (KST)
- 대상 리포지토리: `hololive-bot`
- 범위: `docker-compose.prod.yml` 기반 운영에서 `k8s/` 기반 운영으로 전환한 현재 상태 분석

## 1) 현재 상태 요약

- 기본 배포 모드: `k8s/overlays/prod` (**DNS ExternalName 모드**)
- 대체 배포 모드: `k8s/overlays/prod-ip` (IP 직접 라우팅 모드)
- 외부 의존성 연결 리소스: `EndpointSlice(discovery.k8s.io/v1)`로 정리
- 관측성 스택: Fluent Bit + Loki + Grafana 운영
- 운영 중 테스트 Pod(`*test*`)는 오버레이 패치로 제거 정책 반영

## 2) 장점 (k3s 도입 이득)

1. **복구 자동화/운영 안정성**
   - Pod 비정상 종료 시 자동 재기동, Deployment 롤링 업데이트 지원.
2. **배포 표준화**
   - `kustomize` 오버레이(`prod`, `prod-ip`)로 환경별 구성을 명확하게 관리 가능.
3. **서비스 디스커버리 일관성**
   - Service/DNS 중심으로 내부 통신 규칙 통일.
4. **관측성 확장**
   - 중앙 로그 수집/검색 체계(Loki/Grafana)로 장애 분석 속도 개선.
5. **확장성 확보**
   - 추후 HPA/PDB/NetworkPolicy 등 운영 고도화 기능으로 확장 용이.

## 3) 단점 (k3s 도입 비용)

1. **운영 복잡도 상승**
   - Compose 대비 리소스 종류와 운용 포인트가 많아 학습/관리 비용 증가.
2. **외부 의존성 경계 관리 필요**
   - ExternalName/외부 DNS 대상(IP, 도커 네트워크) 변경 시 드리프트 가능.
3. **단일 노드 환경 한계**
   - 오케스트레이션 이점은 크지만 노드 장애 내성은 제한적.
4. **디버깅 경로 증가**
   - Probe, 롤아웃, 리더십/락(lease) 등 진단 포인트가 Compose보다 많음.
5. **플랫폼 오버헤드**
   - k3s + 로깅 스택 자체의 메모리/운영 오버헤드가 추가됨.

## 4) 핵심 트레이드오프 정리

- **얻는 것**: 운영 표준화, 복구 자동화, 관측성, 확장성
- **치르는 비용**: 복잡도 증가, 외부 의존성 경계 관리 부담, 단일 노드 한계

즉, 현재 전환은 “서비스 운영 품질을 높이는 방향”으로 타당하며,  
성공의 핵심은 **외부 의존성 변경 관리와 운영 자동검증 체계**에 달려 있음.

## 5) 권장 후속 과제 (우선순위)

1. **외부 의존성 드리프트 자동 검증**
   - 배포 전 `externalName/port` 유효성 점검 스크립트 + 체크리스트 운영.
2. **운영 로그 정책 단순화**
   - stdout-only 원칙 강화, hostPath 파일 로그 의존 최소화.
3. **장애 대응 표준화**
   - 공통 진단 런북(rollout/probe/log/lease) 문서화 및 점검 절차 고정.
4. **운영 안전장치 강화**
   - NetworkPolicy/PDB/HPA 도입 타당성 검토 및 단계 적용.
5. **정기 롤백 훈련**
   - `prod ↔ prod-ip` 전환/롤백 리허설 자동화.

## 6) 결론

현재 k3s 전환은 장기 운영 관점에서 이점이 더 큼.  
다만 외부 연동 경계와 운영 자동검증이 약하면 이점이 감소하므로,  
후속 과제 1~3을 우선 실행하는 것이 가장 효과적임.
